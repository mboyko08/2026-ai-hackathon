# Project Plan — CDC Data Quality Validator

## Goal

Build a simple Azure-based data quality validator that compares CDC change data from OLTP source databases against replicated data in a PostgreSQL OLAP warehouse. Source data is generated exclusively by stored procedures.

## Architecture

| Component | Role |
| --- | --- |
| Azure Database for PostgreSQL | OLTP source — generates data via stored procedures |
| Azure SQL Database (SQL Server) | OLTP source — generates data via stored procedures |
| PostgreSQL OLAP Warehouse | Replication target — receives CDC changes |
| CDC / Change Tables | Captures inserts, updates, and deletes from each OLTP source |
| Replication Pipeline | Extracts CDC changes and merges them into the warehouse |
| Centralized Log Area | Aggregates database logs, pipeline logs, and warehouse audit logs |
| Orchestrator Agent | Entry point — identifies scope and coordinates the validation workflow |
| Validation Agent | Builds the validation plan and invokes the CDC Executor skill |
| CDC Executor skill | Runs read-only validation queries (not an autonomous agent) |
| Root Cause Agent | Reads results and logs, identifies failure point, recommends next action |

## Scope

### In Scope

- PostgreSQL OLTP database and stored procedure
- SQL Server OLTP database and stored procedure
- CDC capture or simulation for both sources
- Replication pipeline (extract → stage → merge)
- PostgreSQL OLAP warehouse and merge process
- Centralized logging (database logs, pipeline logs, warehouse audit logs)
- Orchestrator, Validation, and Root Cause agents
- CDC Executor skill
- End-to-end demo with one intentional mismatch

### Out of Scope

- Public API ingestion
- Production-grade CDC framework
- Complex security model
- Automatic remediation without human approval
- Large-scale historical backfills

## Demo Scenario

1. Run PostgreSQL stored procedure to generate OLTP records.
2. Run SQL Server stored procedure to generate OLTP records.
3. Capture or simulate CDC changes in both sources.
4. Run the replication pipeline into the PostgreSQL warehouse.
5. Introduce one intentional mismatch (e.g. skip a CDC batch or create a duplicate key).
6. Ask the Orchestrator Agent to validate the latest CDC replication.
7. Validation Agent selects checks and invokes the CDC Executor skill.
8. CDC Executor skill runs read-only queries and returns structured results.
9. Root Cause Agent reads centralized logs and explains what failed.
10. Produce a validation report with the recommended fix.

## Success Criteria

- Source data generated by stored procedures only
- CDC changes captured or simulated from both OLTP sources
- CDC data replicated into the warehouse
- All six validation checks run successfully
- At least one controlled mismatch detected
- Centralized logs used to explain the root cause
- Clear recommended fix produced
